# Task: surface-pipeline-open-questions

## Description
The user reports that every `design.md` generated by the pipeline opens with a
prose caveat block like:

> Note: `AskUserQuestion` is unavailable inside subagents, so the four open
> decisions below are presented with the agent's recommended option (and the
> alternatives it weighed). Treat the human gate as the open-question step —
> if any recommendation is wrong, redirect there and a revision will re-draft.

`AskUserQuestion` IS declared in the `design-author` agent's `tools:` frontmatter,
and the prompt has a MANDATORY interactive step instructing the agent to call it.
Despite both conditions being true, the agent emits prose rationalizing why it
cannot call the tool. The `questioner` agent exhibits the same failure: during
this pipeline run, the questioner output clarifying questions as prose text
instead of calling `AskUserQuestion`, and the orchestrator had to render them
on its behalf. This is direct evidence the failure is not isolated to
`design-author` — it is a pipeline-wide behavioral pattern.

The user wants: (a) root cause investigation of WHY the model rationalizes
around `AskUserQuestion` even when the tool is wired up, and (b) a fix that
establishes a reusable pattern any pipeline agent can use to surface open
questions interactively, applied first to `design-author` and `questioner`.

## Stated goal
Identify and fix the root cause that causes pipeline subagents to emit prose
question blocks instead of calling `AskUserQuestion`, even when the tool is
declared in their frontmatter and the prompt mandates its use.

## Inferred goal
Establish a reliable, documented pattern for interactive question-asking from
subagents so that the pipeline delivers structured choice prompts to the user
at the two natural interaction points (QUESTION phase and DESIGN phase), and
any future agent that needs user input has a clear, tested pattern to follow.

## Acceptance signals
- The `design-author` calls `AskUserQuestion` during initial dispatch; no prose
  caveat block appears in any generated `design.md`.
- The `questioner` calls `AskUserQuestion` for multi-repo disambiguation and
  clarifying questions rather than emitting prose that the orchestrator must
  re-render.
- A pipeline run through `/team` on a new feature produces a structured
  `AskUserQuestion` interaction at the DESIGN phase with no caveat prose.
- The fix addresses a root cause (agent instruction, tool wiring, or dispatch
  context), not just the symptom (removing the caveat text).
- Any agent that could benefit from the same pattern has a reference
  implementation to follow.

## Open assumptions
- The failure is behavioral (model rationalizing) rather than a wiring bug
  (tool not actually available at runtime); needs verification.
- The pattern should work within the existing agent file format (`tools:`
  frontmatter + markdown prompt) rather than requiring orchestrator-side changes.
- "Apply to others later" means the fix establishes a pattern; it does not
  require updating all 13 agents in this pipeline run.
- The questioner's failure during this very pipeline run is treated as a live
  reproduction of the bug, not a separate issue.
